###: Random port libraries

by Wolfgang Corcoran-Mathe

Abstract

This SRFI proposes a pattern of libraries for binary input ports that produce random bytes. The design leaves the details of random number generation to the implementer and the transformation of bytes to floats, etc. to other libraries.

Issues

Should there be a predicate defining random port types? If so, how should it deal with random ports exported by different libraries? Should it be provided separately?

Rationale

SRFI 27, the best-known (and excellent) proposal for random-number systems in Scheme, includes both too little and too much. On the one hand, it makes no distinction between sources of cryptographic-quality random numbers and sources of the deterministic pseudorandom data sometimes needed for tests & other purposes. On the other, it includes forms for defining new random sources and for parameterizing library procedures over them. It also branches out into procedures for other distributions of random numbers, an area which other proposals (e.g. SRFI 194) have covered in much greater detail.

In this specification I take a different tack. I would like to provide a framework for random-byte generation that adds as little as necessary to the language. To manage this, SRFI ???

  • Uses binary ports, rather than a novel type, for random sources.

  • Uses the Scheme library system--and only the Scheme library system--to distinguish between (hopefully) cryptographic-quality sources and repeatable sources, as well as between different sub-kinds of sources.

  • Specifies no procedures for drawing random data. The random-byte-seeking user is pointed in the direction of read-u8 and read-bytevector, and the other-types-seeking user in the direction of SRFI 194.

Specification

Libraries

A Scheme implementation conforming to this specification provides at least the following R7RS libraries:

  • (srfi ??? random)
  • (srfi ??? random crypto)
  • (srfi ??? random repeatable)

The usual library-name-format variants for these libraries may also be provided:

  • (srfi :??? random crypto), etc.
  • (srfi srfi-??? random crypto), etc.

(srfi ??? random) is simply an alias for (srfi ??? random crypto). (Rationale: Incautious programmers who simply want random numbers get the best available.)

Library protocol

The (srfi ??? random crypto) library and any libraries with the (srfi ??? random crypto) prefix should provide ports that produce cryptographic-quality pseudorandom data.

Both the crypto and repeatable libraries export the make-random-port constructor. repeatable libraries provide an additional interface for creating (pseudo)random ports with predetermined initial states.

Additional libraries

Implementations are encouraged to provide additional libraries (e.g. for difference pseudorandom generation algorithms) under the repeatable "namespace". If they do, they must ensure that each library follows the protocol detailed above. A list of example library names is provided below.

Discussion

This section is informational.

This specification has only one mechanism for choosing between different random sources: each library is tied to a specific generation mechanism.

No means is provided here for getting other kinds of data (integers, reals, etc.) out of random ports. I intend that other libraries (like SRFI 194) provide wrapper procedures for generating those types of data.

If this proposal is incorporated within the R7RS, the suggested library prefix is instead scheme random (e.g. (scheme random crypto), (scheme random repeatable) etc.).

Interface

All libraries

A make-random-port procedure is exported by each library with the (srfi ??? random) prefix.

(make-random-port [initializer]) → binary-input-port

Returns a binary input port from which (pseudo)random bytes can be read, e.g. with read-u8 or read-bytevector.

The optional initializer argument is only used by repeatable libraries and must be a bytevector or binary input port. It is used as follows:

  • If initializer

    is a bytevector of length greater than or equal to random-state-size, then the initial state of the random port returned by make-random-port is derived from this value. This provides a means of creating repeatable random ports which produce the same sequence of random bytes.

  • If initializer is a binary input port, then make-random-port will read data from this port until it has enough to initialize a new port. An error is signalled if enough data cannot be obtained.

repeatable libraries

The following identifiers are exported by all libraries with the (srfi ??? random repeatable) prefix.

random-state-size

A positive exact integer giving the number of bytes needed to describe the state of a random port (of the kind constructed by this library). This can be used to construct a state bytevector for use with make-random-port.

(random-port-state random-port) → bytevector

Returns a bytevector representing the current state of random-port.

Example PRNG library names

This section is informational. Names are based on https://prng.di.unimi.it and Wikipedia.

(srfi ??? random repeatable xorshift)

Library using some kind of xorshift generator.

(srfi ??? random repeatable xoshiro256++)
(srfi ??? random repeatable xoshiro256**)

xoshiro libraries.

(srfi ??? random repeatable mwc256)

Library for a 256-bit multiply-with-carry source.
